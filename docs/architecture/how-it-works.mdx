## Two-phase execution

The engine runs in two phases for every `return get` call.

**Phase 1 — Build the execution plan**

Starting from the set of input nodes you've added (W-2s, 1099s, schedules), the engine does a topological sort of the dependency graph. This determines the order in which nodes must execute.

Multi-instance nodes (you can add multiple W-2s, multiple 1099s) are collected into arrays and passed to their downstream consumers as a batch.

**Phase 2 — Execute in order**

Each node in the plan receives its accumulated inputs, validates them against its Zod schema, runs its `compute()` function, and deposits results into a shared dictionary. Downstream nodes read from that dictionary when their turn comes.

Optional forms that receive no input are simply skipped.

```typescript
// core/runtime/executor.ts (simplified)
const plan = buildExecutionPlan(inputs);  // topo sort

const pending: Record<string, Record<string, unknown>> = {};

for (const instance of plan) {
  const node = registry[instance.nodeType];
  const parsed = node.inputSchema.safeParse(pending[instance.id] ?? {});
  if (!parsed.success) continue;  // optional node with no data

  const result = node.compute(parsed.data);

  for (const output of result.outputs) {
    pending[output.nodeType] = merge(pending[output.nodeType] ?? {}, output.input);
  }
}
```

## Dependency graph

Each node declares which nodes it feeds into via `outputNodeTypes`. The engine uses this to build the graph statically — before any computation runs.

```
w2 ──────────────────────────────────────► f1040
w2 (statutory employee) ─────────────────► schedule_c
w2 (allocated tips) ─────────────────────► form4137
schedule_c ──────────────────────────────► schedule_se → f1040
f1099int ────────────────────────────────► schedule_b → f1040
f1099div ────────────────────────────────► schedule_b → f1040
                                           ► schedule_d → f1040
```

Visualize any part of the graph:

```bash
tax node graph --node_type start
tax node graph --node_type w2 --depth 3
```

## Pure nodes

Every node is a pure function. `compute()` takes validated input, returns outputs. No state, no side effects, no network calls.

This means:
- The same inputs always produce the same outputs
- Nodes are independently testable
- The engine is fully deterministic

## Validation layer

`return validate` runs a separate pass — the IRS MeF business rules engine. This checks cross-field consistency (e.g., total payments must reconcile), filing status requirements, missing required fields, and hundreds of IRS-specific rules.

Validation is separate from computation. A return can compute correctly but still fail validation (e.g., a required signature field is missing).
